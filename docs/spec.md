# Product Spec

## Product Goal

Build a private AI coaching system for one athlete (Sidney Drost) with one immediate objective: help him become the strongest possible natural flat barbell bencher as efficiently, intelligently, and safely as possible.

## Phase 1 Scope

Phase 1 is deliberately narrow.

In scope:

- Bench-specific coaching across all three standards (IPF, gym no wraps, gym + wraps)
- A structured athlete profile with current phase awareness (cut / bulk / peak / pivot)
- Bench training history imported from Strengthlog (1333 sets, Nov 2024 – present)
- A recommendation engine that generates bench sessions (not just follows a plan)
- e1RM tracking per bench standard with trend analysis
- Readiness gating: pain, fatigue, sleep, bodyweight trend
- Deload detection using the 2-of-3 trigger rule
- Progression logic (top single, rep work, back-offs)
- Session autoregulation (safety override when RPE exceeds target)
- Comparable session matching (heavy ↔ heavy, medium ↔ medium)
- Phase-specific programming (cut protocol NOW; year-round builder and peaking later)
- Clear, traceable explanations for every recommendation
- A usable interface for logging, reviewing, and acting on recommendations

Out of scope for now:

- Full-body programming beyond what is strictly needed to support bench decisions
- Accessory exercise prescription (referenced in coach-logic.md but not generated by the engine)
- Nutrition tracking or meal planning
- Generic fitness features
- Broad wellness tracking
- Community, social, or SaaS features
- Multi-user architecture
- Technique video analysis
- Wearable data integration (Apple Watch, Garmin, Whoop — noted for future)

## Target User

One athlete only: Sidney Drost.

The product assumes:

- High motivation and strong interest in optimisation
- Willingness to log: top single, e1RM, hard sets, pain (0–10), sleep, bodyweight
- Desire for specific, non-generic coaching — explain the why
- Familiarity with RPE/RIR, periodisation, and evidence-based training concepts
- Currently in a cut phase (→ 65 kg by Apr 2026), transitioning to year-round builder post-vacation

## What Excellent Looks Like

The coach should feel like:

- A bench specialist who knows Sidney's history, not a generic chatbot
- Recommendations are traceable: "I prescribed X because your e1RM trend shows Y and your pain score is Z"
- Phase-aware: knows the difference between cut-mode and builder-mode programming
- Standard-aware: never conflates IPF paused e1RM with gym TnG e1RM
- Honest about uncertainty: if data is sparse, it says so
- Capable of telling Sidney exactly what to do today and why
- Useful enough to influence real training decisions

## Core Inputs

### Per Session (Minimum)

- Date
- Bench standard used (IPF / gym / gym+wraps — default: gym no wraps)
- Exercise variation (comp paused, TnG, close-grip, Spoto, etc.)
- Top single: weight, RIR
- Working sets: weight × reps × sets, RIR
- Pain score (0–10, by location if applicable)
- Bodyweight (if weighed that day)
- Session type (heavy / medium / light)
- Notes (free text)

### Per Day (Optional but Valuable)

- Sleep hours
- Subjective readiness (1–5 or similar)
- Fatigue/soreness notes

### Profile (Updated Infrequently)

- Current training phase
- Bodyweight target and trend
- Schedule and frequency
- Equipment changes
- Injury/irritation status

## Core Outputs

### Session Recommendation

- Prescribed session type (heavy / medium / light)
- Exercises with sets × reps × target RPE/RIR
- Top single target weight and RPE
- Back-off set weights (calculated from current e1RM and block)
- Rest period guidance
- Warm-up protocol reference

### Status and Trend

- Current e1RM per bench standard (with confidence indicator)
- e1RM trend (up / flat / down) with comparable session basis
- Deload trigger status (which triggers are active, how close to threshold)
- Bodyweight trend and strength-to-bodyweight ratio
- Phase progress (where in the current mesocycle/peak/cut)

### Alerts

- Deload recommendation when 2-of-3 triggers fire
- Safety override notification when RPE exceeds target
- Pain escalation protocol activation
- Strength loss warning thresholds during cut
- Missing data warnings when confidence is reduced

### Explanation

Every recommendation includes:

- Which inputs drove the decision
- Which rule or logic path was applied
- What tradeoff is being made
- What to monitor in the next session

## Candidate Screens or Routes

- Dashboard (e1RM trend, deload status, phase, bodyweight, recent sessions)
- Session recommendation (today's prescribed bench work)
- Session log (input today's results)
- Session review (compare planned vs actual, flag deviations)
- Training history (filterable by date, variation, standard, phase)
- Athlete profile (view/edit profile and phase settings)
- Settings

## Product Principles

- Bench performance comes first
- Personalisation is mandatory — the system must use Sidney's actual data
- Recommendations must be explainable and traceable
- The system generates programming, not just executes a static plan
- Phase 1 stays small and sharp — bench only
- Architecture should allow later expansion without polluting the initial build
- The three bench standards are always tracked independently

## Resolved Questions

- Training data available at start: 1333 bench-related sets from Strengthlog (Nov 2024 – Mar 2026)
- Logging commitment: Sidney logs consistently in Strengthlog and is willing to log the minimum fields
- Technique video analysis: deferred to later phase
- Recovery inputs worth collecting in MVP: pain (0–10), sleep hours, bodyweight
- Bench methodology: encoded directly based on Sidney's reference docs (APTIS, Bench Mastery System), not left generic

## Open Questions

- Implementation stack (deferred until design phase)
- Whether to import/sync from Strengthlog app directly or manual entry
- How to handle the bench standard field for historical data (most is gym no wraps, but not tagged)
- Wearable data integration scope and priority
- Whether the recommendation engine should have a "coach chat" mode or be purely structured output
